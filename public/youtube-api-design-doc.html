1. Application Overview

Kre8Ωr is an AI-native content production operating system built for solo video creators. It manages the full creative pipeline — from idea and script generation through production, post-production, and distribution — for a single creator (Jason Rutland, 7 Kin Homestead, a homesteading content creator with channels on YouTube, TikTok, Instagram, and Lemon8).

The PostΩr module within Kre8Ωr handles multi-platform video distribution. It allows the creator to upload finished video content to YouTube directly from within the Kre8Ωr interface, eliminating the need to switch between tools during the distribution workflow.

Single-creator, private use: Kre8Ωr is a private, password-protected application used exclusively by its developer/owner. It is not a SaaS platform. The YouTube API integration is used solely to publish video content to the creator's own YouTube channel, using credentials belonging to that channel.

3. OAuth 2.0 Implementation

Kre8Ωr implements the standard Google OAuth 2.0 Authorization Code flow for server-side web applications.

Authorization Flow (Step by Step)

1
Initiate: The creator clicks "Connect YouTube" in the PostΩr UI. The server generates a cryptographically random state token (16 hex bytes via crypto.randomBytes) and stores it in the server-side Express session. The user is redirected to Google's authorization endpoint (accounts.google.com/o/oauth2/v2/auth) with response_type=code, access_type=offline, and prompt=consent.
2
Authorize: The creator logs in to their Google account on Google's servers and grants the requested permissions. Kre8Ωr never sees, handles, or stores the user's Google password.
3
Callback: Google redirects to /api/postor/auth/youtube/callback with code and state. The server validates the state against the session value to prevent CSRF attacks. If mismatched or missing, the request is rejected.
4
Exchange: The server exchanges the authorization code for an access token and refresh token via a server-to-server POST to oauth2.googleapis.com/token. The exchange includes the client_secret, which is stored only in the server's .env file, never in the browser.
5
Channel Lookup: Using the new access token, the server calls channels.list?mine=true to retrieve the channel ID and display name.
6
Storage: The access token, refresh token, token expiry timestamp, channel ID, and channel name are stored in the local SQLite database (platform_connections table). Tokens are stored server-side only and are never transmitted to the browser.
7
Token Refresh: Before each upload operation, the server checks if the access token expires within 5 minutes. If so, it automatically refreshes using the stored refresh token via oauth2.googleapis.com/token with grant_type=refresh_token.

Both URIs are registered as Authorized Redirect URIs in the Google Cloud Console OAuth 2.0 client configuration.

4. Video Upload Implementation

Upload Protocol

Kre8Ωr uses the YouTube resumable upload protocol (uploadType=resumable) to upload video files. This protocol is recommended by Google for files larger than 5 MB and provides reliable upload behavior for large video files.

Upload Flow

1
Initiate Resumable Session: A POST request is sent to https://www.googleapis.com/upload/youtube/v3/videos?uploadType=resumable&part=snippet,status with the video metadata (title, description, tags, categoryId) in the request body and the file size in X-Upload-Content-Length. The response contains a Location header with the session upload URI.
2
Upload File: The video file is read from the local filesystem and sent as the request body in a single PUT request to the session upload URI. The Content-Type is set to video/*.
3
Parse Response: On success (HTTP 200/201), YouTube returns the created video resource including its id. The video URL is constructed as https://www.youtube.com/watch?v={id} and stored in the local postor_posts database table alongside the post status.

Metadata Fields Sent to YouTube

Field Part Source in Kre8Ωr Notes
snippet.title snippet Creator-entered in PostΩr UI Required. Max 200 chars.
snippet.description snippet Creator-entered in PostΩr UI Optional.
snippet.tags snippet Creator-entered comma-separated list Optional. Parsed into array.
snippet.categoryId snippet Creator-selected dropdown Default: 22 (People & Blogs).
status.privacyStatus status Creator-selected (Public / Unlisted / Private) Set to private when scheduling.
status.publishAt status Creator-entered datetime (optional) ISO 8601 string. Only sent if scheduling.
status.selfDeclaredMadeForKids status Hardcoded Always false (content is not for children).

Scheduling

When a creator chooses to schedule a video, Kre8Ωr sets status.privacyStatus to "private" and includes status.publishAt with the creator's chosen datetime as an ISO 8601 string. YouTube then natively handles the transition to public at the scheduled time. No polling or additional API calls are required.

5. YouTube Data — Storage, Use, and Handling

Data Received from YouTube API

Data Received From How It Is Stored How It Is Used Shared With Third Parties?
OAuth Access Token Token exchange endpoint Local SQLite database, server-side only. Never sent to browser. Authorizes upload API calls. Auto-refreshed before expiry. No
OAuth Refresh Token Token exchange endpoint Local SQLite database, server-side only. Used to obtain new access tokens without re-authorization. No
Channel ID channels.list Local SQLite database (platform_connections.account_id). Stored for reference. Not currently used in API calls. No
Channel Title channels.list Local SQLite database (platform_connections.account_name). Displayed in the PostΩr UI connection card ("Connected: [Channel Name]") to confirm the correct account is linked. No
Uploaded Video ID Upload response body Local SQLite database (postor_posts.post_id). Used to construct the video URL for display in the Post History UI. No

Data NOT Accessed or Stored

Kre8Ωr does not access, retrieve, or store any of the following YouTube data:

  • Comments, replies, or comment threads
  • Video analytics or performance metrics via YouTube API (analytics are imported separately via CSV export, not the YouTube Analytics API)
  • Playlists, subscriptions, or channel memberships
  • Other users' channel or video data
  • YouTube search results
  • Live streaming data
  • Caption or subtitle data
  • Community posts

Data Retention

OAuth tokens are retained in the local SQLite database indefinitely until the user explicitly disconnects the YouTube account via the Disconnect button in the PostΩr UI (which deletes the platform_connections row). Post history records (video ID, title, URL, status) are retained in postor_posts as a local reference log for the creator.

No data leaves the application. All YouTube API data is stored locally on the creator's machine (or self-hosted server). It is not shared with, transmitted to, or accessible by any third party. The application has no analytics pipeline, telemetry, or external data storage.

6. User Interface — How YouTube Data Is Displayed

Connection Card (PostΩr Page — Platform Connections Section)

After successful authorization, the YouTube platform card displays:

  • Status badge: "Connected" (teal color)
  • Account name: The channel title fetched from channels.list (e.g., "7 Kin Homestead")
  • Disconnect button: Allows the creator to revoke the stored tokens and remove the connection

No thumbnail images, video listings, subscriber counts, or other YouTube content are fetched or displayed.

Post Composer

The creator fills in the following fields before uploading:

  • Video file path (selected from the Kre8Ωr Vault or entered manually)
  • Title
  • Description
  • Privacy (Public / Unlisted / Private)
  • Optional scheduled publish datetime
  • Optional tags and category

All values are creator-entered. No YouTube data is pre-populated or fetched to fill these fields.

Upload Progress (SSE Stream)

During upload, a Server-Sent Events stream delivers progress updates to the browser. A progress bar shows the upload stage (Initiating → Uploading → Done). On completion, a link is displayed: "View →" linking to https://www.youtube.com/watch?v={videoId}. This link is constructed locally from the video ID returned in the upload API response.

Post History Table

A read-only table displays the creator's recent posts including: platform icon, video title, status (Posted / Failed), post date, and a "View →" link to the YouTube URL. All data is sourced from the local postor_posts database table, not from the YouTube API.

7. Privacy & Security Practices

ConcernImplementation
CSRF Protection OAuth state parameter generated with crypto.randomBytes(16), validated server-side against the Express session before any token exchange is performed.
Client Secret Storage GOOGLE_CLIENT_SECRET stored only in server-side .env file. Never exposed to the browser or included in frontend code.
Token Storage Access and refresh tokens stored in server-side SQLite database. Excluded from any browser response. Database file is excluded from version control via .gitignore.
Authentication Guard All /api/postor/* endpoints are behind the application's session authentication guard. Unauthenticated requests receive HTTP 401. The application requires password login before any YouTube API functionality is accessible.
Minimum Scope Only youtube.upload and youtube.readonly scopes are requested — the minimum required for the use case. No management, monetization, or analytics scopes are requested.
Data Isolation No YouTube data is transmitted to external servers, third-party analytics, or logging services. All data remains within the local application environment.
Revocation The creator can disconnect at any time via the UI, which deletes all stored tokens from the database. Tokens can also be revoked directly at myaccount.google.com/permissions.

9. YouTube API Services Terms of Service — Compliance Statement

Kre8Ωr's use of YouTube API Services complies with the YouTube API Services Terms of Service and the YouTube API Services Developer Policies in the following respects:

  • Authorized use only: The YouTube API is used exclusively to upload the creator's own original video content to their own channel and to retrieve their own channel information. No third-party content is accessed or uploaded.
  • No circumvention: The integration does not attempt to circumvent any YouTube functionality, quotas, or access controls. The standard resumable upload protocol is used as documented.
  • Minimum data: Only the data necessary for the described functionality is accessed. No excess scopes are requested.
  • No unauthorized storage: YouTube data (channel name, video ID) is stored only to the extent necessary for the creator to see which account is connected and to provide a post history reference. It is not aggregated, sold, or shared.
  • User control: The creator can disconnect their YouTube account at any time, immediately revoking the application's access to stored tokens.
  • Not used to violate policies: The upload functionality is used only to distribute the creator's own original content produced with the Kre8Ωr pipeline. The application does not automate uploads without creator intent — every upload requires the creator to manually initiate the post action.
